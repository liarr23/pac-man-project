# 吃豆人项目 — 文件分析

## 项目总览

使用 C++ + SFML 开发的吃豆人游戏，采用状态机模式管理游戏流程。

**完成度：约 15%** — 大部分文件为空壳占位，仅 `Ghost.h` 和 `main.cpp` 有实际代码。

## 各文件详细分析

### 核心框架 (`core/`)

| 文件 | 状态 | 职责 | 说明 |
|------|------|------|------|
| `Config.h` | 有代码 | 聚合头文件 | 集中 include 所有模块头文件，其他文件只需引入这一个 |
| `GameEngine.h/.cpp` | 空壳 | 游戏主循环 | 应负责：初始化窗口、驱动游戏主循环（事件处理→更新→渲染）、管理状态切换 |
| `InputHandler.h/.cpp` | 空壳 | 输入处理 | 应负责：捕获键盘/鼠标事件，转化为游戏操作指令（上下左右移动、暂停等） |

### 游戏角色 (`entities/`)

| 文件 | 状态 | 职责 | 说明 |
|------|------|------|------|
| `GameObject.h/.cpp` | **有实现** | 游戏对象基类 | 定义了 `Point` 坐标结构体，`GameObject` 抽象基类（纯虚函数：update、render、getBounds） |
| `Character.h/.cpp` | 有框架 | 角色基类 | 继承 `GameObject`，应封装移动、碰撞检测等通用角色逻辑 |
| `Ghost.h/.cpp` | **有实现** | 幽灵基类 | 定义了方向枚举 `Direction`、幽灵状态枚举 `State`（Chase/Scatter/Frightened/Eaten），含位置、速度、状态切换逻辑，以及一个 `Blinky` 子类雏形 |
| `Blinky.h/.cpp` | 空壳 | 红色幽灵 | 追踪型：直接追向 Pacman 当前位置 |
| `Pinky.h/.cpp` | 空壳 | 粉色幽灵 | 伏击型：瞄准 Pacman 前方 4 格的目标点 |
| `Inky.h/.cpp` | 空壳 | 青色幽灵 | 复杂型：以 Blinky 为参考点做对称计算 |
| `Clyde.h/.cpp` | 空壳 | 橙色幽灵 | 随机型：距离远时追赶，距离近时散开 |
| `Pacman.h/.cpp` | 空壳 | 吃豆人 | 应负责：响应玩家输入移动、吃豆子、碰撞检测、生命管理 |

## 当前问题

1. **`GameObject` 重复定义** — `core/GameObject.h` 和 `entities/GameObject.h` 各有一份，且只有 `entities/` 版本有实际代码
2. **`core/Config.h` 引用错误** — 引用了 `entities/` 目录下的头文件，但使用的是不带路径的 include（如 `#include "Blinky.h"`），编译会找不到
3. **`main.cpp` 使用硬编码绝对路径** — 图片路径写死为 `C:\Users\liar\Desktop\...`，换电脑就会报错
4. **`Ghost.h` 中 `Bilnky` 拼写错误** — 应为 `Blinky`（第 50 行类名 `Bilnky`）
5. **`Ghost` 构造函数参数名冲突** — `speed(speed)` 会把成员变量赋值给自身（第 27 行参数名和成员变量同名）

## 建议的开发优先级

1. **地图系统**（MapManager + Wall）— 没有地图其他东西都没法测试
2. **Pacman 移动**（Pacman + InputHandler）— 核心玩法
3. **豆子系统**（Item + Dot + PowerPellet）— 收集逻辑
4. **幽灵 AI**（Ghost + Blinky/Pinky/Inky/Clyde）— 游戏挑战性
5. **状态管理**（GameState + 各子状态）— 游戏流程
6. **UI 和音效**（UIManager + SoundManager + ScoreManager）— 体验完善
